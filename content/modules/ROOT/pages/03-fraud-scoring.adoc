= Financial Transaction Scoring
:navtitle: Fraud Scoring

== Learning Objectives

* Score financial transactions using real LLM inference combined with rule-based signal detection
* Understand why the response shows both `rule_score` and `llm_score` — and why that matters
* Run compliance checks that use real LLM reasoning, not hardcoded results
* See the A2A protocol in action between agents

== How It Works

The FinServ agent combines two approaches:

[cols="1,2,1",options="header"]
|===
| Approach | What It Does | Latency

| Rule-based signals
| Detects obvious patterns: high amount (>$10K), high-risk country, wire transfer, round numbers
| <1ms

| LLM risk assessment
| granite-2b-cpu evaluates the transaction context and assesses fraud probability (0-100)
| ~500ms

| Combined score
| 60% rule weight + 40% LLM weight → final risk level and recommendation
| —
|===

The response includes both scores separately so you can see what each approach contributed. The model name and inference latency in the response are from the *actual LLM call*, not hardcoded.

== Exercise 1: Score a High-Risk Transaction

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {finserv_url}/api/v1/score-transaction \
  -H "Content-Type: application/json" \
  -d '{
    "transaction_id": "tx-demo-high",
    "amount": 15000.00,
    "country": "NG",
    "merchant_category": "wire_transfer"
  }' | jq .
----

Expected response includes:

* `rule_score`: ~85 (high amount + high-risk country + wire transfer + round number)
* `llm_score`: ~70-80 (LLM agrees this is suspicious)
* `risk_score`: ~83 (combined)
* `risk_level`: "critical"
* `recommendation`: "block"
* `model`: "granite-2b-cpu" (the actual model that ran)
* `inference_ms`: ~500ms (real measured latency)
* `signals`: 5 signals including `llm_risk_assessment` with the LLM's reasoning

== Exercise 2: Score a Low-Risk Transaction

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {finserv_url}/api/v1/score-transaction \
  -H "Content-Type: application/json" \
  -d '{
    "transaction_id": "tx-demo-low",
    "amount": 89.50,
    "country": "US",
    "merchant_category": "retail"
  }' | jq .
----

Expected: `rule_score: 10`, `llm_score: ~5`, `risk_level: "low"`, `recommendation: "approve"`.

Note the LLM still runs — it confirms the low risk. The latency is still ~500ms for the LLM call. The rules are fast (<1ms) but the LLM adds intelligence.

== Exercise 3: Run Compliance Checks

The compliance endpoint also uses real LLM inference — each regulation is evaluated by the model:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {finserv_url}/api/v1/check-compliance \
  -H "Content-Type: application/json" \
  -d '{
    "transaction": {"id": "tx-comp", "amount": 15000, "country": "NG"},
    "regulations": ["aml", "kyc"]
  }' | jq .
----

Each regulation check shows its own `inference_ms` — real LLM reasoning time per check.

== Exercise 4: A2A Agent Discovery

Discover the FinServ agent via its A2A agent card:

[source,bash,role="execute",subs=attributes+]
----
curl -s {finserv_url}/.well-known/agent-card.json | jq '{name, version, protocolVersion, skills: [.skills[].id]}'
----

This is how the orchestrator discovers agents automatically — standard A2A protocol.
