= FinServ Agent
:navtitle: FinServ Agent

*Time: ~15 minutes* · *Terminal*

== The Question

Can LLM inference add intelligence to fraud scoring without replacing fast rule-based systems?

== How It Works

The FinServ agent combines two approaches for every transaction:

[cols="1,2,1",options="header"]
|===
| Approach | What It Does | Latency

| Rule-based signals
| Detects patterns: high amount (>$10K), high-risk country, wire transfer, round numbers
| <1ms

| LLM risk assessment
| granite-2b-cpu evaluates the transaction context and assesses fraud probability (0-100)
| ~500ms

| Combined score
| 60% rule weight + 40% LLM weight = final risk level and recommendation
| —
|===

The response includes both scores separately — you can see what each approach contributed.

== Exercise 1: Score a High-Risk Transaction

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {finserv_url}/api/v1/score-transaction \
  -H "Content-Type: application/json" \
  -d '{
    "transaction_id": "tx-high-risk",
    "amount": 15000.00,
    "country": "NG",
    "merchant_category": "wire_transfer"
  }' | jq .
----

Look at the response:

* `rule_score` vs `llm_score` — the rules and the LLM agree this is suspicious
* `signals` — each signal that fired, including the LLM's reasoning
* `model` and `inference_ms` — real values from the actual LLM call

== Exercise 2: Score a Low-Risk Transaction

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {finserv_url}/api/v1/score-transaction \
  -H "Content-Type: application/json" \
  -d '{
    "transaction_id": "tx-low-risk",
    "amount": 89.50,
    "country": "US",
    "merchant_category": "retail"
  }' | jq .
----

The LLM still runs — it confirms the low risk. The rules fire fast (<1ms) but the LLM adds context. Compare `inference_ms` between this and the high-risk case.

== Exercise 3: Find the Edge Case

Craft a transaction that the rules miss but the LLM catches — or vice versa:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {finserv_url}/api/v1/score-transaction \
  -H "Content-Type: application/json" \
  -d '{
    "transaction_id": "tx-edge",
    "amount": 9999.00,
    "country": "US",
    "merchant_category": "crypto_exchange"
  }' | jq '{rule_score, llm_score, risk_score, recommendation, signals: [.signals[].signal]}'
----

The amount is just under the $10K threshold so the `high_amount` rule doesn't fire — but the LLM may still flag the crypto exchange context. This is why the hybrid approach matters.

== Exercise 4: Run Compliance Checks

Each regulation is evaluated by the LLM independently:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {finserv_url}/api/v1/check-compliance \
  -H "Content-Type: application/json" \
  -d '{
    "transaction": {"id": "tx-comp", "amount": 15000, "country": "NG"},
    "regulations": ["aml", "kyc"]
  }' | jq .
----

Each regulation check shows its own `inference_ms` — real LLM reasoning time per check.

== Exercise 5: Discover the Agent via A2A

[source,bash,role="execute",subs=attributes+]
----
curl -s {finserv_url}/.well-known/agent-card.json | jq '{name, version, protocolVersion, skills: [.skills[].id]}'
----

This is how the orchestrator discovers agents automatically — standard A2A protocol. Try the healthcare agent's card too:

[source,bash,role="execute",subs=attributes+]
----
curl -s {healthcare_url}/.well-known/agent-card.json | jq '{name, skills: [.skills[].id]}'
----

== What You Measured

* Rule-based vs LLM scoring — how they complement each other
* Edge cases where one approach catches what the other misses
* Per-regulation compliance check latency
* A2A agent discovery protocol
