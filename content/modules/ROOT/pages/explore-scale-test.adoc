= Scale Test
:navtitle: Scale Test

*Time: ~15 minutes* · *Terminal*

== The Question

What happens to latency when you fire concurrent requests at a single Xeon 6 node? Cost stays flat — but does throughput?

== Exercise 1: Baseline — Single Request

Time a single pipeline call:

[source,bash,role="execute",subs=attributes+]
----
time curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin."}' \
  | jq .total_ms
----

== Exercise 2: Concurrent Load

Fire 5 concurrent requests and compare:

[source,bash,role="execute",subs=attributes+]
----
for i in $(seq 1 5); do
  curl -s -X POST {healthcare_url}/api/v1/pipeline \
    -H "Content-Type: application/json" \
    -d "{\"text\": \"Record $i: Patient with chest pain and hypertension.\"}" \
    | jq "{record: $i, total_ms}" &
done
wait
----

Compare latency to your baseline. Latency climbs with concurrency — cost stays at $0.

== Exercise 3: Check Pod Resource Usage

[source,bash,role="execute"]
----
oc adm top pods
----

See how CPU utilization scales across the healthcare agent replicas during concurrent load.

== Exercise 4: Aggregate Stats

[source,bash,role="execute",subs=attributes+]
----
curl -s {healthcare_url}/api/v1/stats | jq '{total_requests, avg_latency_ms}'
----

== What You Measured

Throughput is capped by the model serving layer (MAAS), not the agent pods. Adding agent replicas helps latency (~20-30%), but the real throughput gain comes from model serving replicas and INT8 quantization.

The cost line is flat. The efficiency line is what we engineer.
