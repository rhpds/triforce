= The Efficiency Stack
:navtitle: Efficiency Stack

== Learning Objectives

* Understand the seven engineering levers that reduce latency without changing hardware
* See how model selection, conditional pipelines, and MCP tools each contribute
* Run the FP32 vs INT8 model comparison (when optimized models are available)
* Understand the optimization roadmap: what's live, what's tested, what's next

== Why This Matters

The scale test (previous module) showed the tradeoff — latency climbs under load while cost stays flat at $0. The efficiency stack is how you push that latency line down.

Three categories of optimization:

[cols="1,3",options="header"]
|===
| Category | What It Does

| Per-Record Efficiency
| Reduce the work per individual record — use the right model, skip unnecessary steps, use data lookups instead of LLM calls

| Model Optimization
| Same hardware, better inference — quantization, optimized variants, prompt tuning

| Fleet-Scale Throughput
| Scale total output — batch processing, replica scaling, disaggregated inference
|===

== Exercise 1: See Model Selection in Action

The pipeline uses different models for different tasks. Run the pipeline and check which model handled each step:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin and Lisinopril. STEMI with PCI to RCA."}' \
  | jq '.inference_log[] | {node, model, latency_ms}'
----

Notice different models for different steps — the Semantic Router principle applied at the pipeline level.

== Exercise 2: See the Conditional Pipeline

Run with a single-medication patient (no drug interaction check needed):

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "PROGRESS NOTE: 45-year-old female with Hypertension on Amlodipine 10mg. Blood pressure controlled."}' \
  | jq '{entities: [.entities[].text], drug_interactions, inference_log: [.inference_log[].node], total_ms}'
----

With only 1 medication, the `check_interactions` step should be absent from the inference log — the pipeline skipped it.

Now run with multiple medications:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: Patient on Metformin, Lisinopril, Aspirin, and Clopidogrel."}' \
  | jq '{entities: [.entities[].text], drug_interactions: .drug_interactions | length, inference_log: [.inference_log[].node], total_ms}'
----

With 4+ medications, `check_interactions` should appear — and drug interactions should be found.

== Exercise 3: MCP Tool vs LLM Comparison

The drug interaction check uses an MCP tool (FDA database lookup) instead of an LLM call. Compare the latencies:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "Patient on Aspirin 81mg, Clopidogrel 75mg, Metformin 1000mg, and Lisinopril 10mg."}' \
  | jq '.inference_log[] | {node, model, latency_ms}'
----

The `check_interactions` step should show `model: "mcp-tool"` with latency ~15ms — vs the LLM steps at ~500-4000ms. Data lookups don't need inference.

== Exercise 4: Model Comparison (when INT8 available)

When INT8 quantized models are deployed, run the FP32 vs INT8 comparison:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline/compare \
  -H "Content-Type: application/json" \
  -d '{
    "text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin and Lisinopril.",
    "classify_model": "qwen25-3b-int8",
    "ner_model": "granite-2b-int8",
    "summarize_model": "qwen25-3b-int8"
  }' | jq '{baseline_total: .baseline.total_ms, optimized_total: .optimized.total_ms, speedup, delta_ms}'
----

This runs the same pipeline twice — once with FP32 models (baseline) and once with INT8 (optimized) — and shows the real delta.

NOTE: If INT8 models are not yet deployed, this endpoint will return an error for the optimized run. The baseline will still work.

== The Seven Levers

[cols="1,2,1",options="header"]
|===
| Lever | What It Does | Status

| vLLM Semantic Router | Route to right-sized model per request complexity | LIVE
| Conditional Pipeline | Skip inference steps not needed for this record | LIVE
| MCP Tools | Use data lookups instead of LLM calls | LIVE
| Model Selection | Best model per task (qwen25-3b classify, granite-2b NER) | LIVE
| AMQ Streams Batching | Process thousands of records concurrently | LIVE
| INT8 Quantization | Same model, 2-3x faster with AMX instructions | PENDING
| Agent + Model Replica Scaling | Horizontal scaling on existing hardware | TESTED
|===

Each lever compounds. Cost stays at $0/token. Performance is engineered, not purchased.
