= Model Selection
:navtitle: Model Selection

*Time: ~15 minutes* · *Terminal*

== The Question

Does picking the right model per task actually matter? And when should you pay for GPU instead of running on CPU for free?

== Exercise 1: See the Default Model Assignment

Run the pipeline and check which model handled each step:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin and Lisinopril. STEMI with PCI to RCA. Started on Aspirin 81mg."}' \
  | jq '.inference_log[] | {node, model, latency_ms}'
----

Different models for different steps. The pipeline picks the right size for the right task.

== Exercise 2: Override with a Different CPU Model

Try running classify with granite-2b instead of qwen25-3b:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes.", "classify_model": "granite-2b-cpu"}' \
  | jq '{classify_model: .inference_log[0].model, classify_ms: .inference_log[0].latency_ms, total_ms}'
----

Compare the classify latency to Exercise 1. Model selection is a config change, not a code change.

== Exercise 3: Try a GPU Model

Override with a Gaudi GPU model and compare latency and output quality:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin and Lisinopril. Recent STEMI with PCI to RCA. Started on Aspirin 81mg, Clopidogrel 75mg, Atorvastatin 80mg.", "summarize_model": "granite-3-2-8b-instruct"}' \
  | jq '{summarize_model: .inference_log[-1].model, summarize_ms: .inference_log[-1].latency_ms, summary: .summary[:200], total_ms}'
----

Compare to the CPU run: the GPU 8B model may produce a more detailed summary, but check whether the latency and quality difference justify the per-token cost.

== Exercise 4: Benchmark CPU vs GPU Side-by-Side

Use the benchmark endpoint to compare the same task across tiers:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/benchmark/run \
  -H "Content-Type: application/json" \
  -d '{"models": ["granite-2b-cpu", "qwen25-3b-cpu", "granite-3-2-8b-instruct"], "task": "classification", "text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin."}' \
  | jq '.results[] | {model, hardware, latency_ms, output}'
----

Three models, two hardware tiers, same task. For classification, the 2B CPU model is typically fast and accurate — the 8B GPU model doesn't add much. Try it on summarization to see where the bigger model shines:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/benchmark/run \
  -H "Content-Type: application/json" \
  -d '{"models": ["qwen25-3b-cpu", "granite-3-2-8b-instruct"], "task": "summarization", "text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin 1000mg and Lisinopril 10mg. Recent STEMI with PCI to RCA. Started on Aspirin 81mg, Clopidogrel 75mg, Atorvastatin 80mg. EF 45%. Creatinine 1.2. Follow up in 2 weeks."}' \
  | jq '.results[] | {model, hardware, latency_ms, output: .output[:200]}'
----

== Exercise 5: See the Full Model Fleet

[source,bash,role="execute",subs=attributes+]
----
curl -s {healthcare_url}/api/v1/benchmark/models \
  | jq '{cpu: [.cpu_models[] | "\(.model) (\(.cost))"], gpu: [.gpu_models[] | "\(.model) (\(.cost))"]}'
----

9 CPU models at $0/token. 4 GPU models at $/token. The heterogeneous routing module decides per request which tier to use.

== What You Measured

[cols="1,2,2",options="header"]
|===
| Task | CPU Sufficient? | When to Use GPU

| Classification | Yes — 2B/3B CPU models are fast and accurate | Rarely needed
| NER | Yes — granite-2b-cpu optimized for entity extraction | Only for very long documents
| Fraud scoring | Yes — rule signals + 2B LLM covers most cases | Complex multi-factor analysis
| Summarization | Depends on length — 3B handles short, 8B better for complex | Long, multi-section documents
| Complex reasoning | CPU works but slower — phi3-mini at 3.8B | Differential diagnosis, multi-step compliance
|===

The right model per task gives ~5% improvement on CPU alone. Adding GPU for the tasks that need it gives better quality where it matters — without paying GPU prices on tasks that don't.
