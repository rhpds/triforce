= Model Selection
:navtitle: Model Selection

*Time: ~10 minutes* · *Terminal*

== The Question

Does picking the right model per task actually matter? The pipeline uses qwen25-3b-cpu for classification and granite-2b-cpu for NER — because we benchmarked them.

== Exercise: Compare Models Per Step

Run the pipeline and check which model handled each step:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin and Lisinopril. STEMI with PCI to RCA. Started on Aspirin 81mg."}' \
  | jq '.inference_log[] | {node, model, latency_ms}'
----

You should see different models for different steps. The Semantic Router classifies request complexity and routes to the right model — same principle applied at the pipeline level.

== Exercise: Override Models

The pipeline accepts model overrides. Try running classify with granite-2b instead of qwen25-3b:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes.", "classify_model": "granite-2b-cpu"}' \
  | jq '{classify: .inference_log[0], total_ms}'
----

Compare the classify latency to your previous run. Model selection is a config change, not a code change.

== What You Measured

Benchmarked results on this pipeline:

* All granite-2b: ~8.2s total
* All qwen25-3b: ~8.6s (NER slower on long prompts)
* Mixed (qwen classify + granite NER + qwen summarize): ~7.8s

The right model per task gives ~5% improvement with zero hardware change. Model selection is a config change, not a code change.
