= Multi-Model Fusion
:navtitle: Multi-Model Fusion

*Time: ~15 minutes* · *Terminal*

== The Question

For critical decisions — compliance, diagnostics, audit-grade outputs — is one model's answer good enough? Or do you need consensus?

== How It Works

Multi-model fusion sends the same prompt to 3 panel models in parallel. A judge model (8B) then synthesizes the responses — finding consensus, flagging contradictions, and identifying blind spots.

```
Prompt → ┌─ granite-2b-cpu ──→ Response A ─┐
         ├─ qwen25-3b-cpu  ──→ Response B ─┤→ Judge (8B) → Synthesis
         └─ phi3-mini-cpu  ──→ Response C ─┘
```

The panel runs in parallel, so wall-clock time ≈ slowest single model (not 3x). The judge adds ~2s. Total cost: 4 CPU inference calls at $0/token.

== Exercise 1: Run a Fusion Request

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/fusion \
  -H "Content-Type: application/json" \
  -d '{"prompt": "Summarize the key clinical findings for a 72-year-old male with chest pain and diabetes."}' \
  --max-time 60 | jq .
----

NOTE: Fusion runs 4 inference calls (3 panel + 1 judge). Allow up to 60 seconds for the response.

The response includes:

* `panel_responses` — each model's individual answer
* `synthesis` — the judge's merged analysis
* `consensus_points` — where the models agreed
* `contradictions` — where they disagreed
* `total_ms` — end-to-end latency

== Exercise 2: Compare Single vs Fusion

Run the same prompt through a single model, then through fusion:

[source,bash,role="execute",subs=attributes+]
----
echo "=== Single model ==="
curl -s -X POST {healthcare_url}/api/v1/summarize \
  -H "Content-Type: application/json" \
  -d '{"text": "72-year-old male with chest pain, diabetes, on Aspirin and Metformin."}' \
  | jq '{summary: .summary[:100], model, latency_ms}'
----

One model gives one perspective. Fusion gives three perspectives + synthesis. For routine classification, single is fine. For compliance or diagnostics, consensus catches what one model misses.

== The Intel Story

Three small CPU models (2B + 3B + 3.8B) running in parallel on Xeon 6 cost $0 and take the same wall-clock time as a single call. Compare that to running one large GPU model (14B+) — fusion on CPU is cheaper and often more reliable for structured tasks.

The cost equation: cost of being wrong >> cost of 3x inference at $0/token.

== What You Measured

* Fusion latency vs single-model latency (parallel ≈ 1x wall-clock)
* Consensus and contradiction detection across models
* Trade-off: routine tasks don't need fusion; critical decisions do
