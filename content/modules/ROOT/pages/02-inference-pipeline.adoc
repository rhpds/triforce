= Clinical NLP Pipeline
:navtitle: Clinical NLP

== Learning Objectives

By the end of this module, you'll be able to:

* Run a multi-step LangGraph inference pipeline on Intel Xeon 6
* Understand how different models are selected per task (model selection optimization)
* See how the conditional pipeline skips unnecessary inference steps
* Verify every step is tracked in PostgreSQL with the actual model used and real latency

== How It Works

The healthcare agent runs a 4-node LangGraph StateGraph. Different models handle different tasks — the right model for the right job:

[cols="1,2,1",options="header"]
|===
| Step | What It Does | Model

| Classify
| Identify document type (discharge summary, lab report, etc.)
| qwen25-3b-cpu (3B)

| Extract Entities
| Find conditions, medications, procedures
| granite-2b-cpu (2B)

| Check Interactions
| Screen medications for drug-drug interactions
| MCP tool (FDA database — not an LLM call)

| Summarize
| Generate clinical summary with context from prior steps
| qwen25-3b-cpu (3B)
|===

NOTE: The *Check Interactions* step is conditional — it only runs when 2+ medications are detected. If a patient is on one medication, there's nothing to cross-reference. The pipeline skips the step entirely.

NOTE: Drug interaction data comes from a curated FDA / NIH RxNav database via MCP tool — this is a data lookup (~14ms), not an LLM call (~3s). Not everything needs inference.

== Exercise 1: Run the Pipeline via Frontend

. Navigate to the Triforce demo frontend
. In *Act 02: The Proof*, find the clinical text sample
. Click *Run Pipeline on Xeon 6*
. Watch each node light up as it processes:
** *Classify* — the first node activates, shows latency when complete
** *Extract Entities* — NER extraction, longest inference step
** *Check Interactions* — conditional: fires if 2+ medications found
** *Summarize* — generates the clinical summary

[TIP]
====
The latency numbers you see are *real* — they come from the actual LLM inference call on Intel Xeon 6 CPU. The model name shown for each step is extracted from the graph's inference log, not hardcoded.
====

== Exercise 2: Run via API

. Classify a clinical document:
+
[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/classify \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72yo male with Type 2 Diabetes on Metformin 500mg and Lisinopril 10mg. Recent STEMI with PCI to RCA."}' | jq .
----
+
Note the `model` field in the response — it shows the actual model used (from the graph's inference log), not a hardcoded value.

. Run the full pipeline:
+
[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin and Lisinopril. Recent STEMI with PCI to RCA. Started on Aspirin 81mg, Clopidogrel 75mg, Atorvastatin 80mg."}' | jq .
----
+
The response includes:
+
* `classification` — document type
* `entities` — extracted medical entities with types
* `drug_interactions` — interactions found via MCP tool
* `summary` — generated clinical summary
* `inference_log` — per-step latency and actual model used
* `total_ms` — end-to-end pipeline time

. Compare the models used per step in `inference_log`:
+
[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/pipeline \
  -H "Content-Type: application/json" \
  -d '{"text": "DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin and Lisinopril."}' \
  | jq '.inference_log[] | {node, model, latency_ms}'
----

== Exercise 3: Check the Inference Log

Every LLM call is logged to PostgreSQL with the actual model, task type, latency, and accelerator.

[source,bash,role="execute"]
----
oc exec deploy/postgres -n triforce -- psql -U triforce -d triforce -c \
  "SELECT model, task_type, latency_ms, accelerator FROM inference_log ORDER BY created_at DESC LIMIT 10;"
----

Notice: *100% CPU, 0% GPU*. Every inference call ran on Intel Xeon 6.
