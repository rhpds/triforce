└──────────────────────────┬──────────────────────────────────┘
                           │
     ┌─────────────────────┼─────────────────────┐
     │                     │                     │

== Models on MAAS/LiteLLM (Intel Xeon 6 CPU)

[cols="2,1,2",options="header"]
|===
| Model | Parameters | Task in Pipeline

| qwen25-3b-cpu | 3B | Classification, Summarization
| granite-2b-cpu | 2B | NER / Entity extraction
| phi3-mini-cpu | 3.8B | Complex reasoning (fraud analysis, compliance)
|===

All models run on Intel Xeon 6 CPU via MAAS/LiteLLM. No GPU. $0/token.

The vLLM Semantic Router classifies each incoming request by complexity and routes to the right-sized model — simple tasks go to the smallest model, complex reasoning goes to the largest.

== Healthcare Agent Pipeline (LangGraph)

The healthcare agent runs a 4-node pipeline using multiple models:

```
Classify (qwen25-3b)  →  Extract NER (granite-2b)  →  Check Interactions  →  Summarize (qwen25-3b)
                                                          │ (conditional)
                                                          │ skip if <2 meds
                                                          ▼
                                                      MCP Tool
                                                   (FDA database)
```

* **Classify** — Document type (discharge summary, lab report, etc.)
* **Extract NER** — Medical entities (medications, conditions, procedures)
* **Check Interactions** — Drug-drug interactions via MCP tool (database lookup, not LLM)
* **Summarize** — Clinical summary of the record

The conditional step skips the drug interaction check if fewer than 2 medications are found — no point checking interactions with one drug.

== FinServ Agent (Quarkus + LLM)

The financial services agent combines rule-based signal detection with real LLM risk assessment:

* **Rule signals** — Amount thresholds, high-risk countries, transaction categories (<1ms)
* **LLM risk assessment** — granite-2b-cpu evaluates fraud probability (~500ms)
* **Combined score** — 60% rule weight + 40% LLM weight → final risk level

This is NOT a pure rule engine. The LLM call is real, the latency is measured, and the model name in the response comes from the actual inference call.

== MCP Gateway (Tool Federation)

The MCP Gateway federates 8 tools across agents using JSON-RPC 2.0:

* `fhir_patient_lookup` — Patient record retrieval
* `drug_interaction_check` — FDA / NIH RxNav drug interaction database
* `clinical_code_search` — ICD-10 / CPT code search
* `risk_profile_lookup` — Customer risk profile
* `regulatory_rule_check` — Compliance rule evaluation
* `sanction_list_search` — Sanctions screening
* `get_agent_status` — Agent health monitoring
* `get_inference_stats` — Inference telemetry

Not every operation needs an LLM. Drug interaction checking is a database lookup (14ms) vs an LLM call (~3s). The MCP Gateway routes to the right tool.

== Technology Stack

|===
| Component | Technology

| Healthcare Agent | Python 3.11, FastAPI, LangGraph, aiokafka, asyncpg

| Container Runtime | Podman / OpenShift
