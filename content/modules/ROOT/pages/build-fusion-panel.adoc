= Build: Fusion Panel Tuning
:navtitle: Build: Fusion Panel

*Time: ~20 minutes* · *Terminal*

== What You'll Build

You will change which models sit on the fusion panel and which model acts as judge, then measure how panel composition affects quality and latency. By the end, you will have a decision table for when fusion is worth its cost.

Multi-model fusion runs 3 panel models in parallel, then a judge model synthesizes consensus and surfaces contradictions. The system is configured through two environment variables:

* `FUSION_PANEL_MODELS` -- the comma-separated list of models that answer in parallel (default: `granite-2b-cpu,qwen25-3b-cpu,phi3-mini-cpu`)
* `FUSION_JUDGE_MODEL` -- the model that synthesizes the panel's answers (default: `granite-3-2-8b-instruct-cpu`)

== Exercise 1: Run Default Fusion

Run fusion with the default panel (granite-2b, qwen-3b, phi3-mini) judged by granite-8b:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/fusion \
  -H "Content-Type: application/json" \
  -d '{"prompt": "A 72-year-old male with Type 2 Diabetes on Metformin 500mg and Warfarin 5mg presents with acute chest pain radiating to the left arm. Troponin elevated at 8.2 ng/mL. eGFR 38 mL/min. What are the key clinical considerations for immediate management, medication interactions, and renal-adjusted dosing?"}' \
  --max-time 60 | jq '{status, panel: {models: .panel.models, latency_ms: .panel.latency_ms}, judge: {model: .judge.model, latency_ms: .judge.latency_ms, consensus: .judge.consensus, contradictions: .judge.contradictions, blind_spots: .judge.blind_spots}, total_ms}'
----

NOTE: Allow up to 60 seconds -- fusion runs 4 inference calls (3 panel + 1 judge).

== Exercise 2: Swap the Panel

Replace the default 3-small panel with 2 small + 1 medium model. This adds an 8B model to the panel alongside the existing 2B and 3.8B:

[source,bash,role="execute",subs=attributes+]
----
oc set env deploy/healthcare-agent \
  FUSION_PANEL_MODELS="granite-2b-cpu,phi3-mini-cpu,granite-3-2-8b-instruct-cpu"
----

Wait for the rollout:

[source,bash,role="execute",subs=attributes+]
----
oc rollout status deploy/healthcare-agent --timeout=90s
----

== Exercise 4: Design Your Fusion Policy

Not every task needs multi-model consensus. Fusion runs 4 inference calls -- 3 panel + 1 judge -- so it costs 4x a single model call. At $0/token on CPU that cost is pure latency. Decide where the latency is worth it.

== Learning Outcomes

By completing this module, you should now understand:

* Panel composition affects what contradictions and blind spots the judge can find
* A larger model on the panel improves synthesis quality but increases latency
* A smaller judge is faster but may miss nuances in panel disagreements
* Fusion is worth the 4x inference cost only for high-stakes decisions
