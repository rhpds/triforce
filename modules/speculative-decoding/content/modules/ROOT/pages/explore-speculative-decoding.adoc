= Explore: Speculative Decoding
:navtitle: Speculative Decoding

*Time: ~10 minutes* · *Role: ML Engineer, Performance Engineer*

== What You'll Learn

LLM inference generates tokens one at a time — each token waits for the previous one. Speculative decoding breaks this bottleneck by running a small, fast draft model that proposes several tokens ahead, then the target model verifies them in a single pass. Correct tokens are accepted instantly. Wrong ones are regenerated. The output is identical to the target model, and this lab reports the measured latency from the active environment.

== How It Works

```
Standard decoding (sequential):
  Token 1 → Token 2 → Token 3 → Token 4 → Token 5
  3s each = 15s total

Speculative decoding:
  Draft model proposes: [T1, T2, T3, T4, T5] in 200ms
  Target model verifies all 5 in one pass: 3s
  4 correct, 1 rejected → regenerate 1 token: 3s
  Total: measured by the live endpoint
```

The draft model (`granite-350m`) proposes tokens for the `granite-2b-cpu` target. How much this helps depends on prompt shape, sampling settings, and serving hardware, so the demo measures it instead of assuming a fixed multiplier.

== Exercise 1: Compare Draft vs Target Speed

[source,bash,role="execute",subs=attributes+]
----
echo "=== Draft model (granite-350m) ===" && \
curl -s -X POST {healthcare_url}/api/v1/benchmark/run \
  -H "Content-Type: application/json" \
  -d '{"task":"classification","text":"DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin.","models":["granite-350m"]}' \
  | jq '.results[0] | {model, latency_ms, output}' && \
echo "=== Target model (granite-2b) ===" && \
curl -s -X POST {healthcare_url}/api/v1/benchmark/run \
  -H "Content-Type: application/json" \
  -d '{"task":"classification","text":"DISCHARGE SUMMARY: 72-year-old male with Type 2 Diabetes on Metformin.","models":["granite-2b-cpu"]}' \
  | jq '.results[0] | {model, latency_ms, output}'
----

The draft model should be significantly faster while producing the same classification. This speed difference is what speculative decoding exploits — the draft proposes, the target verifies.

== Exercise 3: Measure the Combined Speedup

When speculative decoding is active, the healthcare endpoint measures baseline target latency against the speculative LiteLLM alias:

[source,bash,role="execute",subs=attributes+]
----
curl -s -X POST {healthcare_url}/api/v1/speculative/run \
  -H "Content-Type: application/json" \
  -d '{"task":"summarization","text":"72-year-old male admitted with acute chest pain. History of Type 2 Diabetes, hypertension, CKD stage 3.","max_tokens":128}' \
  | jq '{baseline: .baseline.latency_ms, speculative: .speculative.latency_ms, speedup, message}'
----

If the speculative path is faster, the endpoint reports the measured speedup. If it is not faster on that prompt, the UI says it was configured and measured rather than claiming a speedup.

== The Takeaway

Speculative decoding is invisible to the application. No code changes and no API shape changes are required by callers. The vLLM server handles the draft model and target verification, while Oberon validation decides whether the measured result should be presented as a speedup or simply as configured and measured.
