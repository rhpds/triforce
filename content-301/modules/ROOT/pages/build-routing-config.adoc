= Build: Custom Routing Config
:navtitle: Build: Routing Config

*Time: ~20 minutes* · *Terminal*

== What You'll Build

The semantic router classifies every request into a complexity tier and sends it to the right model -- all driven by a YAML ConfigMap. No code changes required.

You will add a 4th route to the router for high-stakes clinical decisions, test it with real traffic, and tune the scoring weights. By the end, the router will classify text into 4 tiers instead of 3, and you will understand how anchor sentences and scoring weights control routing accuracy.

== Exercise 3: Add a 4th Route

You will add a `critical` route for high-stakes decisions that require the largest CPU model -- situations where the cost of a wrong answer outweighs the cost of extra inference time.

Export the current config for editing:

[source,bash,role="execute",subs=attributes+]
----
oc get configmap semantic-router-config -o jsonpath='{.data.routes\.yaml}' > /tmp/routes.yaml
----

Edit the file to add a 4th route. Open it in your terminal editor, or use the template below as a guide. Add this block after the `complex` route in the `routes:` list:

[source,yaml,subs=attributes+]
----
  - name: critical
    model: granite-3-2-8b-instruct-cpu
    params: "8B"
    description: "High-stakes clinical decisions requiring maximum reasoning depth"
    anchors:
      - "Evaluate whether this patient should be taken off life support given their advance directive and current prognosis"
      - "Determine if this combination of medications poses an immediate life-threatening interaction"
      # Add at least 2 more anchors -- think about what makes a request "critical"
      # Hint: life-or-death decisions, legal liability, irreversible actions
      - "YOUR ANCHOR HERE"
      - "YOUR ANCHOR HERE"
    keywords:
      - "life-threatening"
      - "immediate danger"
      # Add at least 2 more keywords
      - "YOUR KEYWORD HERE"
      - "YOUR KEYWORD HERE"
----

Write your own anchors and keywords. Good anchors are full sentences that represent the kind of request this route should catch. Good keywords are terms that appear in critical requests but not in simple or medium ones.

TIP: The router uses embedding similarity against anchors as the primary signal. Keywords are a fallback when embedding confidence is low. Write anchors that are semantically distinct from the existing routes.

Once you have edited `/tmp/routes.yaml`, apply it:

[source,bash,role="execute",subs=attributes+]
----
oc create configmap semantic-router-config \
  --from-file=routes.yaml=/tmp/routes.yaml \
  --dry-run=client -o yaml | oc apply -f -
----

Restart the router to pick up the new config:

[source,bash,role="execute",subs=attributes+]
----
oc rollout restart deploy/semantic-router && \
oc rollout status deploy/semantic-router --timeout=120s
----

NOTE: The router takes up to 2 minutes to restart. It loads the `all-MiniLM-L6-v2` embedding model into memory and precomputes embeddings for all anchor sentences on startup.

== Exercise 5: Tune Scoring Weights

The scoring section controls how the router combines embedding signals:

* `embedding_max: 0.7` -- weight given to the highest single-anchor similarity
* `embedding_avg: 0.3` -- weight given to average similarity across all anchors

Increasing `embedding_max` makes the router more decisive -- a single strong anchor match dominates. Increasing `embedding_avg` makes it more conservative -- the text must be broadly similar to many anchors.

Edit the scoring weights in `/tmp/routes.yaml`:

[source,yaml,subs=attributes+]
----
scoring:
  weights:
    embedding_max: 0.9
    embedding_avg: 0.1
----

Apply and restart:

[source,bash,role="execute",subs=attributes+]
----
oc create configmap semantic-router-config \
  --from-file=routes.yaml=/tmp/routes.yaml \
  --dry-run=client -o yaml | oc apply -f -
oc rollout restart deploy/semantic-router && \
oc rollout status deploy/semantic-router --timeout=120s
----

== What You Learned

* The semantic router is fully data-driven -- adding a route is a YAML change, not a code change
* Anchor sentences are the primary signal -- the router embeds them and compares via cosine similarity
* Scoring weights control how decisive vs conservative the router is
* A 4th tier lets you reserve expensive models for high-stakes decisions while keeping simple requests on cheap, fast models
* Configuration changes require a pod restart because embeddings are precomputed at startup
